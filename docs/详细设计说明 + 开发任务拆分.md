# 《GPA管理系统 V1》详细设计说明 + 开发任务拆分

## 1. 文档定位

本文档基于现有《软件需求规格说明书（SRS）》与《总体设计说明书》，进一步下沉到**可直接指导编码与测试**的粒度，覆盖核心对象、规则实现、数据库结构、服务层接口、导入规范、规划逻辑、测试重点与开发拆分。系统边界仍严格限定为：**单用户、离线运行、仅处理计入 GPA 的课程、支持百分制与等级制、支持目标 GPA 规划与三情景模拟、不做 OCR/联网抓取/排名预测**。 

---

## 2. 设计总原则

### 2.1 核心设计原则

1. **事实数据与规划数据分离**

   * 真实课程与真实成绩是事实源。
   * 目标 GPA、情景、预期成绩、模拟结果属于规划数据。
   * 规划结果不得覆盖真实成绩。

2. **规则集中封装**

   * 百分制/等级制转绩点、GPA 计算口径、目标可达性判定都收敛到规则模块。
   * 业务服务层不散落硬编码公式。

3. **派生结果尽量动态计算**

   * 当前 GPA、模拟 GPA、已计入学分、可达性结论原则上动态算出。
   * 仅在需要保留“历史规划快照”时存储规划结果。

4. **统一课程池**

   * 全部课程进入同一课程集合。
   * 通过 `status = 已修 / 未修` 区分。
   * 不拆“已修表 / 未修表”。

## 3.2 Course

### 3.2.1 职责

表示一门课程的**基础事实信息**，不直接保存当前 GPA，不承载情景模拟数据。

### 3.2.2 建议字段

| 字段       | 类型         | 必填 | 说明                                   |
| ---------- | ------------ | ---: | -------------------------------------- |
| id         | UUID/String  |   是 | 主键                                   |
| courseName | String       |   是 | 课程名称                               |
| semester   | String       |   是 | 所属学期，格式要求年份+季节，如2026春  |
| credit     | Decimal(5,2) |   是 | 学分，必须 > 0                         |
| status     | Enum         |   是 | `COMPLETED` / `PLANNED`                |
| scoreType  | Enum         |   否 | `PERCENTAGE` / `GRADE`；无成绩时可为空 |
| note       | String       |   否 | 备注                                   |
| createdAt  | Datetime     |   是 | 创建时间                               |
| updatedAt  | Datetime     |   是 | 更新时间                               |

### 3.2.3 关键约束

* `status = PLANNED` 时，课程不能有真实成绩。
* `status = COMPLETED` 时，允许暂时没有成绩。
* `credit` 必须为正数。
* `semester` 必填，且格式需要统一。
* **说明**：V1 中课程默认只对应一种成绩表达方式，因此 `scoreType` 暂放在课程表；后续若扩展多次成绩记录，再考虑迁移到成绩记录表。

## 3.3 ScoreRecord

### 3.3.1 职责

表示某门课程的**真实成绩事实**与其规则换算结果。

### 3.3.2 建议字段

| 字段               | 类型           | 必填 | 说明                |
| ---------------- | ------------ | -: | ----------------- |
| id               | UUID/String  |  是 | 主键                |
| courseId         | UUID/String  |  是 | 外键，唯一关联一门课程       |
| hasScore         | Boolean      |  是 | 是否已有成绩            |
| actualScoreRaw   | String       |  否 | 原始成绩值，例如 `92`、`优` |
| actualGradePoint | Decimal(4,3) |  否 | 由规则换算后的绩点         |
| calculatedByRule | String       |  否 | 规则标识，例如 `BUAA_V1` |
| updatedAt        | Datetime     |  是 | 更新时间              |

### 3.3.3 说明

* `actualScoreRaw` 存原始值，保证可追溯。
* `actualGradePoint` 在写入/修改成绩时由规则计算得到，并存库；当前 GPA 等汇总结果动态计算。
* `hasScore = false` 时，`actualScoreRaw`、`actualGradePoint` 必须为空。
* 一门课程最多一个真实成绩记录。

### 3.4.1 职责

表示用户一次明确发起的 GPA 规划目标。

### 3.4.2 建议字段

| 字段                        | 类型           | 必填 | 说明              |
| ------------------------- | ------------ | -: | --------------- |
| id                        | UUID/String  |  是 | 主键              |
| targetGpa                 | Decimal(4,3) |  是 | 目标 GPA          |
| basedOnCurrentGpa         | Decimal(4,3) |  是 | 发起规划时的当前 GPA 快照 |
| basedOnCompletedCreditSum | Decimal(6,2) |  是 | 发起规划时已计入 GPA 学分 |
| feasible                  | Boolean      |  否 | 目标是否理论可达        |
| infeasibleReason          | String       |  否 | 不可达原因           |
| createdAt                 | Datetime     |  是 | 创建时间            |

### 3.4.3 说明

该对象用于保留“某次规划”的上下文快照，避免后续数据变化导致历史规划无从解释。

## 3.5 PlanningScenario

### 3.5.1 职责

表示某个目标下的一种情景：乐观 / 中性 / 保守。

### 3.5.2 建议字段

| 字段                      | 类型           | 必填 | 说明                                        |
| ----------------------- | ------------ | -: | ----------------------------------------- |
| id                      | UUID/String  |  是 | 主键                                        |
| targetId                | UUID/String  |  是 | 外键                                        |
| scenarioType            | Enum         |  是 | `OPTIMISTIC` / `NEUTRAL` / `CONSERVATIVE` |
| simulatedFinalGpa       | Decimal(4,3) |  否 | 模拟最终 GPA                                  |
| requiredFutureAverageGp | Decimal(4,3) |  否 | 达成目标所需未来平均绩点                              |
| createdAt               | Datetime     |  是 | 创建时间                                      |

### 3.5.3 说明

* 一个 `PlanningTarget` 固定对应三个 `PlanningScenario`。
* `requiredFutureAverageGp` 对三个场景可相同，因为它由目标反推，不由场景决定。
* `simulatedFinalGpa` 则由场景下的预期成绩决定。

### 3.6.1 职责

表示某个情景下，对某门未修课程设置的预期成绩。

### 3.6.2 建议字段

| 字段                 | 类型           | 必填 | 说明        |
| ------------------ | ------------ | -: | --------- |

| courseId           | UUID/String  |  是 | 外键，关联未修课程 |
| expectedScoreRaw   | String       |  否 | 预期成绩原始值   |
| expectedGradePoint | Decimal(4,3) |  否 | 预期绩点      |
| createdAt          | Datetime     |  是 | 创建时间      |
| updatedAt          | Datetime     |  是 | 更新时间      |

### 3.6.3 关键约束

* 只能引用 `status = PLANNED` 的课程。
* 不得引用已删除课程。
* 不得写回真实成绩表。

## 3.7 RuleProvider / RuleEngine

## 3.7.1 RuleProvider

### 职责

向系统提供当前启用的 GPA 规则对象。

### 接口建议

```python
class RuleProvider(Protocol):
    def get_rule_id(self) -> str: ...
    def get_grade_point_rule(self) -> GradePointRule: ...
    def get_gpa_computation_rule(self) -> GpaComputationRule: ...
```

## 3.7.2 RuleEngine

### 职责

执行规则计算，而不是保存业务数据。

### 接口建议

```python
class RuleEngine(Protocol):
    def convert_to_grade_point(self, score_type: ScoreType, raw_score: str) -> Decimal: ...
    def calculate_current_gpa(self, courses: List[CourseWithScore]) -> GpaSummary: ...
    def calculate_required_future_average(
        self,
        target_gpa: Decimal,
        completed_quality_point_sum: Decimal,
        completed_credit_sum: Decimal,
        future_credit_sum: Decimal
    ) -> RequiredFutureResult: ...
```

### 设计结论

* `RuleProvider` 负责“提供规则”；
* `RuleEngine` 负责“执行规则”；
* 未来更换学校规则时，只需替换 Provider/Engine 实现，而不是改业务服务。

## 4. GPA 规则详细设计

## 4.1 百分制映射处理方式

百分制课程绩点公式为：

$$
GP = 4 - \frac{3(100-X)^2}{1600}, \quad 60 \le X \le 100
$$
并且：

* `X < 60` 时，绩点 = `0`
* `X = 100` 时，绩点 = `4`
* `X = 60` 时，绩点 = `1`

### 4.1.1 计算函数建议

```text
if X is null -> 非法
if X < 0 or X > 100 -> 非法
if X < 60 -> 0
else -> 4 - 3 * (100 - X)^2 / 1600
```

### 4.1.2 精度建议

* 内部计算使用 `BigDecimal`
* 保留至少 4 位小数计算
* 展示时统一四舍五入到 3 位小数

---

## 4.2 等级制映射处理方式

根据图片规则，五级分制对应如下：

| 等级  |  绩点 |
| --- | --: |
| 优   | 4.0 |
| 良好  | 3.5 |
| 中等  | 2.8 |
| 及格  | 1.7 |
| 不及格 | 0.0 |

### 4.2.1 存储建议

* 数据库存原始等级字符串，例如 `优`、`良好`
* 规则层统一映射为绩点
* 不做 UI 自定义

### 4.2.2 非法等级

以下均应判定非法：

* 空字符串
* 未在枚举集合中的值
* 与百分制混填的数据，例如 `85` 被声明为等级制

---

## 4.3 GPA 计算流程

SRS 已明确 GPA 为加权平均：
**GPA = Σ(绩点 × 学分) / Σ学分**，且仅纳入已修、已有有效成绩、可换算绩点的课程。

### 4.3.1 具体流程

1. 查询全部课程
2. 过滤 `status = COMPLETED`
3. 左连接 `ScoreRecord`
4. 保留 `hasScore = true` 且 `actualGradePoint != null`
5. 逐门计算课程学分绩点：

   * `courseQualityPoint = actualGradePoint * credit`
6. 累加：

   * `qualityPointSum = Σ(courseQualityPoint)`
   * `creditSum = Σ(credit)`
7. 若 `creditSum = 0`

   * 当前 GPA 返回 `null` 或 `0`，建议返回 `null + “暂无可计算课程”`
8. 否则：

   * `gpa = qualityPointSum / creditSum`

### 4.3.2 输出结构建议

```python
@dataclass
class GpaSummary:
    current_gpa: Decimal
    counted_credit_sum: Decimal
    counted_course_count: int
    quality_point_sum: Decimal
    items: List[CourseGpaItem]
```

---

## 4.4 对非法成绩、空成绩、未修课程的处理逻辑

## 4.4.1 非法成绩

### 规则

* 不允许保存
* 服务层直接返回校验错误
* 不进入数据库
* 不触发 GPA 重算

## 4.4.2 空成绩

### 适用场景

* 已修课程但尚未出分

### 处理

* `Course.status = COMPLETED`
* `ScoreRecord.hasScore = false`
* 不参与 GPA 分子和分母计算

## 4.4.3 未修课程

### 处理

* 不允许存在真实成绩
* 不参与当前 GPA
* 可以进入规划模块，保存预期成绩

---

## 4.5 哪些值存库，哪些值动态计算

## 4.5.1 建议存库

| 数据      | 是否存库 | 原因             |
| ------- | ---- | -------------- |
| 课程基础信息  | 是    | 事实数据           |
| 原始成绩值   | 是    | 事实数据           |
| 实际绩点    | 是    | 规则换算结果，便于查询与追溯 |
| 目标 GPA  | 是    | 规划输入           |
| 各情景预期成绩 | 是    | 规划输入           |
| 历史规划快照  | 是    | 便于恢复和解释        |

## 4.5.2 建议动态计算

| 数据       | 原因            |
| -------- | ------------- |
| 当前 GPA   | 派生值，基础数据变化即失效 |
| 已计入学分    | 派生值           |
| 已计入课程数   | 派生值           |
| 模拟最终 GPA | 派生值           |
| 理论可达性    | 派生值           |
| 所需未来平均绩点 | 派生值           |

### 结论

**真实输入存库，统计与推导结果动态计算。**

## 5. 规划模块详细设计

## 5.1 目标 GPA 倒推公式

设：

* `Cg` = 当前已计入 GPA 的总学分绩点和
* `Cc` = 当前已计入 GPA 的总学分
* `Fc` = 未修课程总学分
* `T` = 目标 GPA
* `Rg` = 为达成目标所需未来课程平均绩点

目标满足：

$$
\frac{Cg + Rg \times Fc}{Cc + Fc} = T
$$
可得：

$$
Rg = \frac{T(Cc + Fc) - Cg}{Fc}
$$


### 5.1.1 边界处理

* `Fc = 0`：无法规划未来，直接返回不可规划
* `Rg < 0`：理论上无需未来课程支撑，目标已达成
* `Rg > 4.0`：按当前规则不可达
* `0 <= Rg <= 4.0`：理论可达

---

## 5.2 模拟 GPA 计算逻辑

### 5.2.1 输入

* 当前真实已修课程 GPA 基线
* 某一情景下所有未修课程的 `expectedGradePoint`

### 5.2.2 公式

设：

* `Pg` = 当前真实课程总学分绩点和
* `Pc` = 当前真实课程学分总和
* `Eg` = 情景中所有未修课程预期学分绩点和
* `Ec` = 情景中所有未修课程学分总和

则：

$$
SimulatedFinalGpa = \frac{Pg + Eg}{Pc + Ec}
$$


### 5.2.3 缺少预期成绩时的处理

这里不能擅自模糊，否则结果不稳定。建议明确规则：

* 对某情景中**未设置预期成绩**的未修课程：

  * **默认不参与模拟**，并在结果中标记“该情景未覆盖全部未修课程”
  * 不建议强行自动填默认值

### 5.2.4 更稳的实现

同时输出两个指标：

1. `coveredPlannedCredit`：本情景已设置预期成绩的学分数
2. `isFullCoverage`：是否覆盖全部未修课程

这样前端能明确告诉用户：这只是部分模拟还是完整模拟。

---

## 5.3 乐观 / 中性 / 保守情景如何组织

## 5.3.1 数据组织方式

一个 `PlanningTarget` 固定生成三个 `PlanningScenario`：

| scenarioType | 含义 |
| ------------ | -- |
| OPTIMISTIC   | 乐观 |
| NEUTRAL      | 中性 |
| CONSERVATIVE | 保守 |

每个情景下有各自独立的 `ScenarioCourseExpectation` 列表。

## 5.3.2 设计理由

* 同一门未修课程可以在不同情景下有不同预期成绩
* 情景之间互不污染
* 支持分别计算与分别展示

## 5.3.3 是否需要系统内置默认成绩

V1 不应擅自内置复杂算法。建议只做两种模式：

### 模式A：纯手动

用户逐门填三种情景的预期成绩

### 模式B：批量复制后微调

* 用户先填写“中性”
* 系统允许复制到“乐观/保守”
* 再手工改少量课程

这属于 UI 层便捷能力，不改变数据模型。

---

## 5.4 规划数据与真实数据如何隔离

## 5.4.1 物理隔离

* 真实数据：`course`、`score_record`
* 规划数据：`planning_target`、`planning_scenario`、`scenario_course_expectation`

## 5.4.2 服务隔离

* `ScoreService` 只操作真实成绩
* `PlanningService` 只操作预期成绩
* 两边通过 `courseId` 关联，但不能跨表覆盖字段

## 5.4.3 查询隔离

* 当前 GPA 查询只读真实成绩
* 模拟 GPA 查询 = 真实基线 + 规划预期成绩
* 不允许“把预期成绩当真实成绩去 join 进当前 GPA”

### 结论

这是整个系统最关键的数据安全边界之一。

## 6. 导入模块详细设计

## 6.1 推荐的标准导入文本格式

建议采用**行式 KV 结构**，原因是：

* 人工可读
* AI 容易生成
* 解析成本低
* 出错定位容易

### 6.1.1 推荐格式

```text
course_name=离散数学; semester=2024春; credit=3.0; status=已修; score_type=百分制; score=92
course_name=大学英语; semester=2024春; credit=2.0; status=已修; score_type=等级制; score=良好
course_name=操作系统; semester=2025秋; credit=4.0; status=未修
```

### 6.1.2 字段说明

| 字段          |   必填 | 说明        |
| ----------- | ---: | --------- |
| course_name |    是 | 课程名       |
| semester    |    是 | 学期        |
| credit      |    是 | 学分        |
| status      |    是 | 已修/未修     |
| score_type  | 条件必填 | 已修且有成绩时必填 |
| score       | 条件必填 | 已修且有成绩时必填 |

### 6.1.3 为什么不建议 JSON 作为 V1 主导入格式

不是不能做，而是没必要：

* 对普通用户不友好
* 错误提示更难定位
* AI 生成 JSON 时更容易多层嵌套出错

---

## 6.2 字段校验规则

## 6.2.1 基础字段校验

| 字段        | 校验规则                            |
| ----------- | ----------------------------------- |
| course_name | 非空，去首尾空格后长度 > 0          |
| semester    | 匹配统一格式，`YYYY季节`            |
| credit      | 数值型，且 > 0                      |
| status      | 只能是 `已修` 或 `未修`             |
| score_type  | 若存在，只能是 `百分制` 或 `等级制` |

## 6.2.2 状态联动校验

| 条件                 | 规则                         |
| ------------------ | -------------------------- |
| status=未修          | 不允许出现 score_type / score   |
| status=已修 且无成绩     | 允许省略 score_type / score    |
| status=已修 且有 score | 必须同时有 score_type 与合法 score |

## 6.2.3 百分制成绩校验

* 必须可解析为数字
* 范围 `[0,100]`

## 6.2.4 等级制成绩校验

* 只能在固定集合中：

  * `优`
  * `良好`
  * `中等`
  * `及格`
  * `不及格`

---

## 6.3 冲突检测规则

这是一个真正需要明确的工程点。

## 6.3.1 课程唯一性判定

**`course_name + semester`**作为业务唯一键。
V1 假设同一学期内不存在需要区分的同名不同课程，故采用 `course_name + semester` 作为业务唯一键。

## 6.3.2 冲突类型

| 类型   | 说明                | 建议处理                         |
| ---- | ----------------- | ---------------------------- |
| 完全重复 | 课程名、学期相同，且字段一致    | 跳过并报告 `DUPLICATE_SAME`       |
| 主键冲突 | 课程名、学期相同，但学分/状态不同 | 拒绝该条并报告 `DUPLICATE_CONFLICT` |
| 成绩冲突 | 已有成绩与导入成绩不一致      | 拒绝该条并报告 `SCORE_CONFLICT`     |
| 状态冲突 | 已有课程为未修，导入却给真实成绩  | 拒绝该条并报告 `STATUS_CONFLICT`    |

### 结论

V1 不做自动合并，不静默覆盖，采用**保守拒绝策略**。

## 6.4 错误报告格式建议

建议导入结果返回如下结构：

```python
@dataclass
class ImportResult:
    total_lines: int
    success_count: int
    skipped_count: int
    failed_count: int
    issues: List['ImportIssue']
```

```python

    line_number: int
    raw_line: str
    error_code: str
    field_name: Optional[str]
    message: str
    severity: str # ERROR / WARNING
```

### 6.4.1 错误码建议

| errorCode                | 含义       |
| ------------------------ | -------- |
| MISSING_REQUIRED_FIELD   | 缺少必填字段   |

| SCORE_CONFLICT           | 成绩冲突     |

### 6.4.2 返回示例

## 10. 单元测试设计重点

## 10.1 GPA 计算测试点

## 10.1.1 百分制映射

* 100 分映射 4.0
* 60 分映射 1.0
* 59 分映射 0
* 0 分映射 0
* 90、80、70 等中间值计算正确
* 非法输入：-1、101、空字符串、字母

## 10.1.2 等级制映射

* 优 → 4.0
* 良好 → 3.5
* 中等 → 2.8
* 及格 → 1.7
* 不及格 → 0
* 非法等级：A、B+、pass、null

## 10.1.3 GPA 汇总

* 单门课 GPA
* 多门不同学分加权
* 已修无成绩课程不参与
* 未修课程不参与
* 全部课程都无有效成绩时返回空结果

---

## 10.2 导入校验测试点

* 缺 `course_name`
* 缺 `semester`
* 学分为负数/0/非数字
* `status` 非法值
* `status=未修` 但带 `score`
* `status=已修` 有 `score` 但无 `score_type`
* 百分制成绩越界
* 等级制成绩非法
* 同一文本内部重复行
* 与数据库已有课程冲突
* 完全重复记录是否被正确跳过

---

## 10.3 规划逻辑测试点

* 当前无未修课程时创建规划
* 目标 GPA 已经达成
* 目标 GPA 不可达（所需未来平均绩点 > 4）
* 所需未来平均绩点刚好 = 4
* 某情景仅覆盖部分未修课程
* 三情景下同一课程不同预期成绩能独立计算
* 修改真实成绩后，规划基线同步变化
* 修改课程学分后，requiredFutureAverageGp 正确变化

---

## 10.4 状态联动测试点

* 未修课程禁止录真实成绩
* 已修课程允许无成绩
* 课程状态从未修改为已修后，原 expectation 是否清理
* 删除课程后，score_record 是否级联删除
* 删除课程后，scenario expectation 是否级联删除
* 成绩修改后，当前 GPA 是否立即更新
* 成绩修改后，规划结果立即刷新
