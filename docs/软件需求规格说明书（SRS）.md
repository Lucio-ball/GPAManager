# GPA管理系统 V1《软件需求规格说明书（SRS）》

## 1. 引言

### 1.1 项目背景

GPA（Grade Point Average，平均绩点）是大学生衡量学业表现的重要指标之一，直接影响奖学金评定、推免资格、出国申请以及个人学习规划。现实中，学生通常通过纸面记录、表格软件或零散笔记管理课程成绩，存在以下问题：

* 课程信息分散，难以统一维护；
* GPA 计算依赖手工或外部表格，容易出错；
* 难以基于当前成绩状态，系统性评估未来课程成绩要求；
* 对未来学期的目标绩点规划缺乏可视化和可模拟的支持。

因此，需要一个**面向大学生个人使用的、单用户、离线运行的 GPA 管理系统**，用于统一管理课程、学分与成绩信息，自动计算当前 GPA，并支持围绕目标 GPA 的未来成绩规划。

### 1.2 项目目标

本项目拟开发一个 GPA 管理系统 V1，满足以下目标：

* 支持用户维护计入 GPA 的课程信息；
* 支持录入和修改课程成绩；
* 支持基于学校既定规则自动计算当前 GPA；
* 支持基于目标 GPA 对未修课程进行成绩规划与模拟；
* 支持通过结构化文本进行批量导入；
* 保证系统在**单用户、离线环境**下可稳定使用。

### 1.3 术语定义

| 术语      | 定义                                     |
| ------- | -------------------------------------- |
| GPA     | Grade Point Average，平均绩点，按“加权平均绩点”方式计算 |
| 绩点      | 按学校既定规则由课程成绩换算得到的数值                    |
| 学分      | 课程的权重值，用于 GPA 加权计算                     |
| 学期      | 课程所属修读学期，用于课程记录、展示、筛选与导入校验           |
| 已修课程    | 用户已完成修读的课程，包括已出成绩和暂未出成绩两种情况            |
| 未修课程    | 尚未修读或尚未完成的课程，用于未来规划                    |
| 百分制     | 以 0–100 分表示成绩的评分方式                     |
| 等级制     | 以 A、B、C 等等级表示成绩的评分方式                   |
| 目标 GPA  | 用户希望在某一阶段达到的总体 GPA 目标值                 |
| 情景规划    | 按乐观、中性、保守三种预设情景，对未来课程成绩进行模拟            |
| 结构化文本导入 | 用户先通过外部 AI 将原始材料整理成标准文本，再粘贴至系统中导入      |

## 2. 系统概述

### 2.1 产品定位

GPA 管理系统 V1 是一个**面向大学生个人用户的离线 GPA 管理工具**。其核心价值在于：

* 将课程、学分、成绩信息集中管理；
* 自动化完成 GPA 计算；
* 为未来课程成绩安排提供量化规划依据。

本系统不是教务系统，不承担官方成绩管理职责，也不提供多用户协作、院校推荐、保研预测等扩展服务。

### 2.2 用户特征

本系统目标用户具有以下特征：

* 为单个大学生个人用户；
* 具有基础的课程、学分、成绩概念认知；
* 希望长期跟踪自己的绩点变化；
* 愿意手动录入或粘贴整理后的结构化数据；
* 不要求系统联网或跨设备同步。

### 2.3 运行环境

V1 系统运行环境要求如下：

* **用户类型**：单用户；
* **运行方式**：离线运行；
* **网络要求**：正常使用时不依赖网络；
* **数据存储**：本地存储；
* **输入设备**：键盘、鼠标；
* **适用平台**：桌面端个人计算机。

> 说明：本 SRS 仅约束软件需求，不在此阶段限定具体开发语言、框架或数据库实现方式。

### 2.4 约束条件

系统需满足以下约束：

1. 仅处理**计入 GPA 的课程**；
2. 课程统一纳入同一课程集合，通过“已修/未修”字段区分状态；
3. 支持百分制和等级制两种成绩表示方式；
4. V1 仅按学校既定 GPA 规则计算；
5. V1 需为未来 GPA 规则扩展预留接口，但不实现完整规则引擎；
6. V1 不支持 OCR、截图/PDF 自动解析、教务系统自动抓取；
7. V1 不处理补考、缓考、缺考等复杂成绩规则。

### 2.5 批判性评估

**优势：**

* 需求边界明确，适合课程项目开发；
* 聚焦个人核心痛点，功能闭环完整；
* 离线单用户架构降低复杂度，利于快速落地。

**劣势（风险）：**

* 不直接对接教务系统，数据录入仍依赖用户；
* 学校 GPA 规则若后续变化，V1 需要手工调整；
* 未覆盖复杂成绩制度，适用范围故意收窄。

**置信度评级：高**
理由：以上内容均直接来自已确认需求基线，需求边界清晰。

## 3. 功能需求

### 3.1 课程管理

#### 3.1.1 功能说明

系统应支持用户对课程基础信息进行新增、查看、修改和删除管理。

#### 3.1.2 功能要求

1. 用户可以新增课程；
2. 用户可以编辑已有课程信息；
3. 用户可以删除课程；
4. 用户可以查看课程列表；
5. 每门课程必须标识为“已修”或“未修”；
6. 课程仅允许录入**计入 GPA** 的课程；
7. 每门课程应记录所属学期；
8. 对于已存在课程，系统应避免无意义重复录入；
9. 删除课程后，系统应同步更新 GPA 与规划结果。

#### 3.1.3 输入项

课程名称、所属学期、学分、修读状态、成绩类型等。

#### 3.1.4 输出项

包含所属学期信息的课程列表、单课程详情、更新后的 GPA 结果、更新后的规划结果。

---

### 3.2 成绩管理

#### 3.2.1 功能说明

系统应支持对已修课程录入、修改和查看成绩信息，并在相关数据变化后自动重算。

#### 3.2.2 功能要求

1. 已修课程允许录入成绩；
2. 已修课程允许暂时无成绩；
3. 成绩支持百分制和等级制；
4. 用户可以修改已录入成绩；
5. 成绩修改后系统应自动重新计算 GPA；
6. 未修课程不得录入“最终已得成绩”，但可录入“预期成绩”用于规划。

#### 3.2.3 输入校验要求

1. 百分制成绩必须位于合法区间内；
2. 等级制成绩必须属于系统支持的等级集合；
3. 已修课程若录入成绩，则成绩类型与成绩值必须匹配；
4. 无成绩状态应与“已修”状态兼容；
5. 非法输入时系统应提示并拒绝保存。

---

### 3.3 GPA 计算

#### 3.3.1 功能说明

系统应根据学校既定规则，对已修且已有有效成绩的课程自动计算当前 GPA。

#### 3.3.2 功能要求

1. GPA 计算公式为：
   **GPA = Σ(绩点 × 学分) / Σ学分**
2. 仅纳入：

   * 计入 GPA 的课程；
   * 已修课程；
   * 已存在有效成绩且可换算绩点的课程。
3. 暂无成绩的已修课程不参与当前 GPA 分子与分母计算；
4. 成绩、学分、课程状态变化后，系统应自动重算 GPA；
5. 系统应展示：

   * 当前 GPA；
   * 已计入计算的总学分；
   * 已计入计算的课程数量。

#### 3.3.3 输出要求

系统至少应输出以下结果：

* 当前 GPA；
* 当前已计入 GPA 的学分总数；
* 当前已计入 GPA 的课程明细。

---

### 3.4 目标规划

#### 3.4.1 功能说明

系统应支持用户输入目标 GPA，并基于当前已修课程和未修课程进行未来成绩规划与模拟。

#### 3.4.2 功能要求

1. 用户可以输入目标 GPA；
2. 系统应基于当前 GPA 基础、未修课程学分总量，倒推未来课程平均要求；
3. 用户可以为未修课程录入预期成绩；
4. 系统应根据未修课程预期成绩模拟最终 GPA；
5. 系统应支持三种情景：

   * 乐观
   * 中性
   * 保守
6. 系统应分别输出各情景下的模拟结果；
7. 当目标 GPA 在当前课程条件下不可达时，系统应给出明确提示；
8. 当未修课程不足以支撑目标 GPA 时，系统应说明不可行原因。

#### 3.4.3 规划输出

系统应至少输出：

* 目标 GPA；
* 为达成目标所需的未来平均绩点或平均成绩要求；
* 各情景下的模拟最终 GPA；
* 目标是否可达；
* 不可达时的原因说明。

#### 3.4.4 边界说明

1. V1 仅做总体 GPA 目标规划；
2. 不做排名预测；
3. 不做保研概率预测；
4. 不做院校推荐；
5. 不引入复杂优化算法或选课推荐逻辑。

---

### 3.5 批量导入

#### 3.5.1 功能说明

系统应支持用户通过标准格式的结构化文本批量导入课程与成绩数据。

#### 3.5.2 功能要求

1. 用户可将结构化文本粘贴至系统；
2. 系统应对文本格式进行校验；
3. 校验通过后，系统应完成导入；
4. 导入失败时，系统应给出错误提示；
5. 导入成功后，系统应自动更新课程数据与 GPA 结果；
6. 批量导入应支持课程基础字段、状态字段、成绩字段。

#### 3.5.3 校验要求

系统至少应校验：

* 必填字段是否存在；
* 学期字段是否合法；
* 学分是否合法；
* 已修/未修状态是否合法；
* 百分制/等级制字段是否匹配；
* 同名课程重复导入时的处理是否合法。

> 注：V1 可采用“拒绝导入冲突项并提示用户修正”的策略，但具体冲突处理界面不在本 SRS 中展开。

---

### 3.6 AI 辅助导入说明

#### 3.6.1 功能定位

V1 不直接集成 AI 服务，也不直接解析截图、PDF 或教务原始页面。
AI 在本系统中的作用仅体现在**导入前置环节**：

* 用户先借助外部 AI 工具，将原始文件整理为系统规定的结构化文本，系统可以提供相关提示词，便于用户借助外部 AI 工具；
* 用户再将该文本粘贴到系统中；
* 系统负责格式校验与导入。

#### 3.6.2 系统责任边界

系统负责：

* 接收结构化文本；
* 校验其格式合法性；
* 将其转化为内部课程数据。

系统不负责：

* OCR 识别；
* PDF/截图直接解析；
* 教务系统页面抓取；
* 外部 AI 调用与结果质量保证。

## 4. 业务规则

### 4.1 百分制/等级制换算

1. 百分制成绩应按学校既定规则换算为绩点；
2. 等级制成绩应按学校既定规则换算为绩点；
3. V1 中换算规则固定配置，不提供用户界面级自定义；
4. 系统内部应保留规则扩展能力，以支持后续版本增加自定义规则。

> 说明：由于当前需求基线未给出具体换算表，本 SRS 仅规定“必须按学校既定规则实现”，具体换算表在详细设计阶段固化。

### 4.2 加权 GPA 计算规则

1. GPA 使用加权平均计算；
2. 权重为课程学分；
3. 公式为：
   **GPA = Σ(绩点 × 学分) / Σ学分**
4. 仅对满足条件的课程进行计算；
5. 无有效成绩课程不计入当前 GPA。

### 4.3 学期字段规则

1. 每门课程应记录所属学期；
2. 学期字段用于课程记录、展示、筛选、排序与导入校验；
3. 学期字段本身不改变 GPA 计算公式与换算规则；
4. 系统应支持在课程列表或课程详情中保留该字段信息。

### 4.4 已修/未修逻辑

1. 所有课程均属于统一课程集合；
2. “已修/未修”用于区分课程当前状态；
3. 已修课程可无成绩；
4. 未修课程不参与当前 GPA 计算；
5. 未修课程可参与目标规划；
6. 已修课程在无成绩时不参与 GPA 计算，但保留在课程记录中。

### 4.5 边界规则

1. 不计入 GPA 的课程不进入本系统计算范围；
2. 补考、缓考、缺考等复杂规则 V1 不处理；
3. 若课程无有效成绩映射，则不得计入 GPA。

### 4.6 批判性评估

**优势：**

* 规则简洁，逻辑闭环；
* 适合离线单用户数据模型；
* 学期字段增强了课程组织能力，且不增加核心计算复杂度；
* 为后续规则扩展预留接口但不提前复杂化。

**劣势（风险）：**

* 当前未给出具体换算表，详细设计阶段必须补齐；
* 学期字段的格式规范若不统一，可能影响筛选体验与导入一致性；
* 课程唯一标识设计若不清晰，可能影响重复数据判断与导入一致性。

**置信度评级：中高**
理由：总体规则已确认，但换算表与课程唯一标识细节仍需在后续设计阶段固化。

## 5. 非功能需求

### 5.1 正确性

1. 系统对 GPA 的计算结果必须与学校既定规则一致；
2. 数据变更后重算结果必须正确；
3. 导入校验结果必须与实际数据格式一致；
4. 规划结果必须基于当前系统内数据实时计算，不能使用过期缓存结果。

### 5.2 易用性

1. 用户应能在较少操作步骤内完成课程新增、成绩录入与 GPA 查看；
2. 输入错误时应提供清晰提示；
3. 导入失败时应提示具体问题位置或问题类型；
4. 规划结果应以易理解方式展示，不应仅输出公式或中间变量。

### 5.3 可维护性

1. 系统应将 GPA 规则、数据管理、规划逻辑模块化；
2. 学校 GPA 规则调整时，应能局部修改；
3. 代码结构应便于后续增加自定义规则功能；
4. 数据字段应具备清晰命名和统一约束。

### 5.4 离线可用性

1. 系统核心功能必须在无网络环境下可用；
2. 本地已有数据应可在离线状态下正常读取与使用；
3. 系统不得依赖在线接口完成 GPA 计算与规划。

### 5.5 性能

1. 对个人课程规模的数据集，系统应能在可接受时间内完成导入与重算；
2. 课程数据变更后，GPA 重算应及时响应；
3. 规划计算应在用户可感知范围内快速返回结果。

> 说明：由于 V1 为单用户个人工具，本阶段不对并发性能提出要求。

## 6. 数据项说明

## 6.1 课程字段

| 字段名           | 含义   | 类型约束 | 必填 | 说明                |
| ------------- | ---- | ---- | -- | ----------------- |
| course_name   | 课程名称 | 文本   | 是  | 用于标识课程            |
| course_semester | 所属学期 | 枚举 | 是  | 用于标识课程所属学期        |
| course_credit | 学分   | 数值   | 是  | 必须为合法正数           |
| course_status | 修读状态 | 枚举   | 是  | 已修 / 未修           |
| score_type    | 成绩类型 | 枚举   | 否  | 百分制 / 等级制；无成绩时可为空 |
| course_note   | 备注   | 文本   | 否  | V1 可选，不参与核心计算     |

## 6.2 成绩字段

| 字段名                | 含义        | 类型约束  | 必填 | 说明                |
| ------------------ | --------- | ----- | -- | ----------------- |
| actual_score       | 实际成绩值     | 文本或数值 | 否  | 已修课程可为空           |
| actual_grade_point | 实际绩点      | 数值    | 否  | 由换算规则得到           |
| has_score          | 是否已有成绩    | 布尔/逻辑 | 是  | 区分“已修无成绩”与“已修有成绩” |

## 6.3 规划相关字段

| 字段名                     | 含义     | 类型约束  | 必填 | 说明           |
| ----------------------- | ------ | ----- | -- | ------------ |
| target_gpa              | 目标 GPA | 数值    | 否  | 用户输入的规划目标    |
| expected_score          | 预期成绩   | 文本或数值 | 否  | 仅用于未修课程规划    |
| scenario_type           | 情景类型   | 枚举    | 否  | 乐观 / 中性 / 保守 |
| simulated_gpa           | 模拟 GPA | 数值    | 否  | 基于预期成绩计算     |
| required_future_average | 未来平均要求 | 数值    | 否  | 为达目标反推结果     |
| target_feasible         | 是否可达   | 布尔/逻辑 | 否  | 目标可达性判断      |
| infeasible_reason       | 不可达原因  | 文本    | 否  | 当不可达时填写      |

### 6.4 数据一致性要求

1. 课程状态为“未修”时，不应存在实际已得成绩；
2. 每门课程必须存在合法的学期字段；
3. 课程状态为“已修”且无成绩时，`has_score` 应为否；
4. 课程状态为“已修”且有成绩时，成绩类型和值必须匹配；
5. GPA 计算仅使用合法且有效的课程数据；
6. 规划计算应明确区分“实际成绩”和“预期成绩”。

| UC-04 | 计算当前 GPA  | 系统根据已有课程成绩计算 GPA  |
| UC-05 | 目标 GPA 规划 | 用户输入目标 GPA 并进行规划  |
| UC-06 | 修改成绩后自动重算 | 用户修改成绩后系统自动更新 GPA |

---

### 7.2 UC-01 新增课程

**参与者：** 用户
**前置条件：** 系统已启动并可正常操作
**后置条件：** 新课程被保存到课程列表中

**基本流程：**

1. 用户进入课程新增界面；
2. 输入课程名称、所属学期、学分、修读状态等信息；
3. 系统校验输入合法性；
4. 系统保存课程；
5. 系统更新课程列表。

**异常流程：**

* 学分非法，系统提示错误并拒绝保存；
* 学期字段非法或缺失，系统提示错误并拒绝保存；
* 课程信息不完整，系统提示补全；
* 课程重复冲突，系统提示用户处理。

---

### 7.3 UC-02 录入成绩

**参与者：** 用户
**前置条件：** 已存在一门“已修”课程
**后置条件：** 成绩被成功保存，并参与 GPA 计算

**基本流程：**

1. 用户选择某门已修课程；
2. 输入成绩类型和成绩值；
3. 系统校验成绩合法性；
4. 系统保存成绩并换算绩点；
5. 系统自动重算 GPA；
6. 系统显示更新后的 GPA。

**异常流程：**

* 成绩与成绩类型不匹配；
* 百分制成绩超出范围；
* 等级制成绩不在允许集合中。

---

### 7.4 UC-03 批量导入数据

**参与者：** 用户
**前置条件：** 用户已准备好结构化文本
**后置条件：** 合法数据导入系统

**基本流程：**

1. 用户进入导入界面；
2. 粘贴结构化文本；
3. 系统执行格式校验；
4. 校验通过后导入课程和成绩数据；
5. 系统更新课程列表与 GPA 结果。

**异常流程：**

* 文本格式错误；
* 字段缺失；
* 学期字段非法；
* 学分或成绩非法；
* 存在重复或冲突数据。

---

### 7.5 UC-04 计算当前 GPA

**参与者：** 系统（用户触发查看）
**前置条件：** 存在已修且有有效成绩的课程
**后置条件：** 当前 GPA 被正确展示

**基本流程：**

1. 系统读取全部课程；
2. 筛选已修且有有效成绩的课程；
3. 按既定规则换算绩点；
4. 依据加权公式计算 GPA；
5. 输出当前 GPA、计入学分和课程明细。

---

### 7.6 UC-05 目标 GPA 规划

**参与者：** 用户
**前置条件：** 系统中存在当前课程数据与未修课程数据
**后置条件：** 返回目标规划结果

**基本流程：**

1. 用户输入目标 GPA；
2. 系统读取当前已修课程与未修课程；
3. 系统反推未来平均要求；
4. 用户可为未修课程输入预期成绩；
5. 系统按乐观、中性、保守情景分别模拟；
6. 系统输出最终模拟 GPA 与可达性结果。

**异常流程：**

* 目标 GPA 输入非法；
* 未修课程为空，无法进行未来规划；
* 目标超出可实现范围，系统提示不可达。

---

### 7.7 UC-06 修改成绩后自动重算

**参与者：** 用户
**前置条件：** 某课程已有成绩记录
**后置条件：** 成绩被更新，GPA 自动重算

**基本流程：**

1. 用户修改某课程成绩；
2. 系统保存新的成绩值；
3. 系统重新换算绩点；
4. 系统自动重算 GPA；
5. 系统刷新显示最新 GPA 和规划结果。

**异常流程：**

* 修改后的成绩非法；
* 数据保存失败。

## 8. V1 范围与非范围

### 8.1 V1 范围

V1 版本明确包括以下内容：

1. 课程信息管理；
2. 已修/未修状态管理；
3. 百分制与等级制成绩录入；
4. 基于学校规则的 GPA 自动计算；
5. 成绩修改后的自动重算；
6. 基于目标 GPA 的未来成绩规划；
7. 乐观/中性/保守三种情景模拟；
8. 结构化文本批量导入；
9. 导入格式校验。

### 8.2 V1 非范围

V1 版本明确不包括以下内容：

1. 多用户支持；
2. 管理员角色；
3. 云同步；
4. 联网功能；
5. 教务系统自动抓取；
6. OCR；
7. 直接解析截图/PDF；
8. 排名预测；
9. 保研概率预测；
10. 院校推荐；
11. 补考/缓考/缺考等复杂成绩规则；
12. 完整 GPA 规则自定义引擎；
13. 选课建议与智能优化推荐。

### 8.3 边界判断原则

凡不属于“课程管理—成绩管理—GPA 计算—目标规划—结构化导入”主链路的需求，V1 原则上不纳入实现范围，避免项目失焦。

**优势：**

* 范围可控，便于课程项目按期完成；
* 功能链路完整，能支撑演示与验收；
* 有助于后续迭代扩展。

**劣势（风险）：**

* 用户可能期待更自动化的数据采集；
* 非范围压缩较多，短期体验可能不如全功能产品。

**置信度评级：高**
理由：该范围划分与用户给定边界一致，没有越界扩展。

## 9. 验收标准草案

### 9.1 功能验收标准

#### 9.1.1 课程管理

* 能新增课程；
* 能编辑课程；
* 能删除课程；
* 课程可记录所属学期；
* 能区分已修/未修；
* 删除课程后相关结果同步更新。

#### 9.1.2 成绩管理

* 已修课程可录入成绩；
* 已修课程允许无成绩存在；
* 支持百分制和等级制；
* 成绩修改后可保存；
* 修改后 GPA 自动重算。

#### 9.1.3 GPA 计算

* 系统按学校既定规则换算绩点；
* 系统按学分加权公式计算 GPA；
* 暂无成绩课程不纳入当前 GPA。

#### 9.1.4 目标规划

* 用户可输入目标 GPA；
* 系统可倒推未来平均要求；
* 用户可录入未修课程预期成绩；
* 系统可输出乐观/中性/保守三种情景模拟结果；
* 当目标不可达时，系统能明确提示。

#### 9.1.5 批量导入

* 用户可粘贴结构化文本；
* 系统可校验导入格式；
* 合法文本可成功导入；
* 学期字段异常时会被拒绝并提示原因；
* 非法文本会被拒绝并提示原因。

---

### 9.2 非功能验收标准

1. 系统在离线状态下可完成核心功能；
2. 常规个人数据规模下，操作响应正常；
3. 输入错误时有明确提示；
4. 主要模块职责清晰，便于后续维护。

---

### 9.3 数据正确性验收标准

1. 对给定课程样例数据，系统输出 GPA 与人工核算一致；
2. 修改成绩后，系统输出结果与重新人工核算一致；
3. 导入前后课程总数、学期、学分与成绩信息一致；
4. 目标规划结果与公式计算一致。

---

### 9.4 文档一致性验收标准

1. 需求范围与需求基线一致；
2. 所有核心功能均有明确输入、处理、输出；
3. 所有关键业务规则均可追踪到功能与验收项；
4. 不出现超出 V1 范围的强制实现要求。

---

## 10. 结论

本 SRS 初稿围绕 GPA 管理系统 V1 的核心目标，已形成以下闭环：

* **课程数据管理闭环**：新增、修改、删除、查看；
* **成绩管理闭环**：录入、修改、校验、重算；
* **GPA 计算闭环**：规则换算、加权计算、结果展示；
* **目标规划闭环**：目标输入、未来倒推、情景模拟、可达性判断；
* **导入闭环**：外部 AI 整理、结构化文本粘贴、系统校验导入。

**整体结论：该 SRS 已具备继续推进总体设计的基础。**
